Preliminaries

Before we start, let's wave goodbye to TauFST, the custom web prolog we have been using for FSTs, and let's say hello to SWISH, the web prolog we will use for grammars! The prolog language doesn't have built-in support for FSTs, so we used a custom prolog for our FSTs. But prolog does have built-in support for a type of grammar called Definite Clause Grammars (DCGs), and we will use this to create Context-Free Grammars (CFGs).

We don't need to worry about what exactly DCGs are. (But for those who are curious, here's a link with a brief introduction: http://www.learnprolognow.org/lpnpage.php?pagetype=html&pageid=lpn-htmlse29).) All we need to know is that if we have a context-free grammar rule like

S â†’ NP VP

(where S stands for sentence, NP stands for noun phrase, and VP stands for verb phrase), we can write the rule in a prolog file this way:

s --> np, vp.

Note the period at the end, the comma between the symbols in the right-hand side, and the two dashes that form the arrow. The arrow must look like the one above, there must always be a period at the end, and there must be comma between symbols on the right-hand side.

In class, we talked about the symbols in the alphabet of the language (the symbols used to make the strings), and the symbols that appear in the diagram but are not in the alphabet of the language. With CFGs, we refer to the symbols in the alphabet as terminal symbols (used to form strings in the language) and those other symbols as non-terminal symbols (used in internal structure). The rule above deals only with non-terminals. Rules with terminal symbols are similar, but we will use square brackets around the terminal symbols, as in the example below:

n --> [dogs].

Again, from the two rules above, we can see that dogs is a terminal symbol (a symbol in the alphabet of the language) that can be used in strings in the language, and NP and VP are non-terminal symbols, which are not used in strings.

Let's try that using SWISH, which allows us to run SWI Prolog online. Open the following page using your web browser:

https://swish.swi-prolog.org

On the left side of the window, there are two buttons: Program and Notebook. Click the Program button, and you'll get a prolog editor. From here, things are similar to what we saw with TauFST. There's a place to write a program (where it says "Your prolog rules and facts go here..."), there is a place to write queries (where it says "Your query goes here..."), and right above it there's a place where the results will appear.

Try typing this in the program area:

voiced(aa).
voiced(iy).

Then type this in the query area:

voiced(X).

To run your query, press the blue button that says "Run!" (or you can use ctrl-Enter).

In the results area, you'll see X=aa. To get more results, instead of pressing Enter again as we did in TauFST, we click the "Next" button. Notice that next to the next button there are also buttons for 10, 100 and 1000. You can use those to see the next 10, 100 or 1000 answers.

Now, let's try this with a CFG.

Delete everything in the program area, and type this:

s --> a, b.
a --> [they].
b --> [run].

This is a simple CFG that defines a language with just one string: they run. To check if "they run" is in the language, we use this query:

s([they, run], []).

(Here, we are considering every string that has a tree with s at the very top to be a string in the language, and that's where the s in the query comes from.)

To see everything in the language, we can use this query:

s(X, []).

(X is just a prolog variable.)

Now we are ready to write CFGs in prolog!

Part 2: Duck at and the trees

  
Now that we have seen how to use DCGs to determine whether strings are in the language defined by a CFG, let's see how to get the corresponding parse trees. To get parse trees from prolog DCGs, we can add an argument to each non-terminal that corresponds to the tree. The arguments of the non-terminals on the left-hand side of the rules tell us what subtree is created by that rule. For example, the s rule above becomes:

  
s(s(X, Y)) --> np(X), vp(Y).

  
which is the same as

  
s(s(Subtree1, Subtree2)) --> np(Subtree1), vp(Subtree2).

  
Here we see that the non-terminal symbol on the left-hand side is s. The argument of s (the stuff inside the parentheses) tells us that by applying this rule we are forming a subtree with s at the root, with two children, and each of these children is a subtree that we get from the two items on the right-hand side of the rule. Subtree1 is the subtree we get from the np, and Subtree2 is the subtree we get from the vp.

  
Modify the grammar from part 1 to produce trees by using rules with the format above.

  
Use the template below get started with part 2. Again, notice the table statements, and make sure all of the non-terminals are listed.

:- table np/3.

:- table pp/3.

% enter your lexical rules

Part 3: The garden behind the house in the garden

  
Starting with your grammar for part 2, add additional lexical rules for the following words: barn (n) and behind (p). Use the resulting grammar to parse the following sentences:

  
the dogs chased the cats

  
the dogs chased the cats in the garden

  
the dogs chased the cats in the garden behind the barn

  
the dogs chased the cats in the garden behind the barn in the garden

  
the dogs chased the cats in the garden behind the barn in the garden behind the barn

  
Notice that the number of possible trees increases rapidly with the number of prepositional phrases (PPs). To see how many parse trees there are for a sentence, we can use just a little bit of prolog. Try this:

  
findall(T, s(T, [the,dogs,chased,the,cats,in,the,garden], []), L), length(L, N).

  
The findall/3 predicate gives us all of the solutions for a variable (in this case, T), and puts them in a list (in this case, L). Then, the length/2 predicate tells us the number of elements in a list. In the example above, N is the number of elements in L.

  
Create a table (just a regular table in a Word doc or a Google doc) that shows the number of trees when we have 0, 1, 2, 3 and 4 PPs in a VP (as in the example sentences above). Can you identify the sequence that relates the number of PPs and the number of trees? (It is a known, named sequence, which can easily be found by typing the numbers on google.)

FAQ

Q: When I use my grammar, I get a message that says something like "Clauses of np/2 are not together in the source-file". Is this an error?

  
A: That message is not an error. It is a warning that you have rules with the same left-hand side that don't appear one right after the other in your grammar. That is completely fine. If you want to stop these messages, you can just group all of your np rules, all your vp rules, etc. Alternatively, you can type this before your grammar rules:

  
:- discontiguous np/2.

  
(and do the same for vp, etc.)

  
But don't worry. The bottom line is that these messages don't affect how your grammar works.
